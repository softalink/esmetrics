# esmalert (vmalert port) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Port VictoriaMetrics `vmalert` to a standalone Rust `esmalert` daemon that evaluates alerting + recording rules against esmetrics, sends alerts to Alertmanager, and persists alert state across restarts — running existing vmalert rule files unchanged.

**Architecture:** Two new workspace crates — `esm-gotemplate` (a faithful Go-`text/template` subset engine) and `esmalert` (the daemon) — plus a prompb WriteRequest **encoder** added to the existing `esm-protoparser`. Sync stack only: hand-rolled `esm-http` server + blocking `reqwest`, no tokio. esmalert is a separate process that talks HTTP to esmetrics as datasource and remote-write target.

**Tech Stack:** Rust (edition 2021, rust-version 1.85), `esm-http`, `reqwest` (blocking), `serde_yaml` (strict), `serde_json`, `snap` (snappy), `regex` (RE2), `esm-metricsql` (expr validation), `esm-protoparser` (prompb), `esm-common` (metrics/HTTP helpers).

## Porting Convention (read before every task)

This is a **faithful port**. For each task the authoritative behavioral source is the cited upstream file at `/home/test/refsrc/VictoriaMetrics/app/vmalert/<path>` (pinned at v1.146.0 per the repo `UPSTREAM` file). The plan gives you: the exact Rust interfaces to produce, real failing tests to satisfy, and the subtle semantics that are easy to get wrong. When the plan says "port `<file>:<lines>`", read that Go code and translate it faithfully — preserve metric names, label names, wire formats, validation messages' intent, and numeric behavior. Do **not** invent behavior the upstream does not have, and do **not** add flags/features marked deferred in the spec.

Reference existing in-repo ports for idiom: `crates/esmauth` (sync HTTP daemon, CLI flags, auth), `crates/esm-protoparser` (prompb decode — you add encode), `crates/esm-metricsql` (MetricsQL parse API), the repo's Go-`'g'`-format helper (search `grep -rn "fn format_g\|'g' format\|fmt_g" crates/`).

## Task 4: `esm-gotemplate` executor

**Files:**
- Create: `crates/esm-gotemplate/src/value.rs`, `crates/esm-gotemplate/src/exec.rs`
- Test: inline in `exec.rs`

**Interfaces:**
- Consumes: `ast::*`.
- Produces (in `value.rs`):
  - `pub struct Metric { pub labels: std::collections::BTreeMap<String,String>, pub value: f64, pub timestamp: i64 }`
  - `pub enum Value { Nil, Bool(bool), Int(i64), Float(f64), Str(String), Metric(Metric), Vec(Vec<Metric>), Map(std::collections::BTreeMap<String, Value>), List(Vec<Value>) }`
  - `impl Value` helpers: `fn truthy(&self) -> bool`, `fn render_string(&self) -> String` (Go `fmt`-style default rendering; floats via the repo's Go-`'g'` helper).
- Produces (in `exec.rs`):
  - `pub type FuncFn = Box<dyn Fn(&[Value]) -> Result<Value, TemplateError> + Send + Sync>`
  - `pub struct Funcs(pub std::collections::HashMap<String, FuncFn>)`
  - `pub fn exec_nodes(nodes: &[Node], dot: &Value, root: &Value, funcs: &Funcs) -> Result<String, TemplateError>` (internal engine; public `Template::render` wraps it in Task 7).

**Reference:** Go `text/template/exec.go`. Critical semantics to preserve:
- **`missingkey=zero`**: indexing a `Map` with an absent key yields `Value::Nil` (renders as empty), never an error.
- `if`/`with`: empty value → else branch. `with` rebinds dot to the pipeline value on the truthy branch.
- `range` over `Vec`/`List`/`Map`: rebinds dot per element; binds `$v`/`$i,$v` when declared; empty collection → else branch.
- `$var` lexical scope: a `HashMap<String,Value>` stack; entering range/with pushes a scope, exiting pops it; `$` is the root value.
- Pipelines: evaluate cmd 0, feed its result as the **last** argument of the next `Call`, left to right.
- Field chain `.A.B`: index Map/Metric-label successively; missingkey=zero at each hop.
